# ADR №001: Выбор фреймворка для Telegram-бота

**Статус:** Принято

**Контекст:**  
Мы начинаем разработку Telegram-бота на Python. Основные требования: асинхронность, удобство работы с современными возможностями Bot API (клавиатуры, FSM, middleware), активное сообщество и долгосрочная поддержка. На рынке представлены три основных кандидата:  
- **python-telegram-bot (PTB)** – синхронный (с поддержкой asyncio через PTB v20+), богатый функционал, долгая история.  
- **aiogram** – изначально асинхронный, версия 3.x полностью переписана на asyncio, тесная интеграция с FSM, middleware, удобная документация.  
- **pyTelegramBotAPI (telebot)** – простой синхронный фреймворк, меньше возможностей для сложных сценариев.

**Рассмотренные варианты:**

1. **python-telegram-bot (v20+)**  
   - *Плюсы:* широкая известность, обилие примеров, поддержка asyncio с недавних пор.  
   - *Минусы:* архитектура всё ещё хранит legacy синхронных версий, middleware реализованы не так гибко, как в aiogram; переход на asyncio произошёл недавно, меньше оптимизированных решений.

2. **aiogram 3**  
   - *Плюсы:* изначально асинхронный, максимально использует возможности asyncio; встроенный FSM на базе Redis или памяти; мощная система middleware; отличная документация; родная поддержка всех типов клавиатур и API методов.  
   - *Минусы:* меньше примеров под версию 3 (но быстро растущее сообщество).

3. **pyTelegramBotAPI**  
   - *Плюсы:* простота, низкий порог входа.  
   - *Минусы:* синхронность, ограниченные возможности для сложной логики, отсутствие встроенного FSM, слабая поддержка middleware.

**Решение:**  
Выбираем **aiogram 3**. Его асинхронная архитектура идеально ложится на наш стек (PostgreSQL через asyncpg, Redis через aioredis), а встроенные механизмы FSM и middleware ускорят разработку и повысят качество кода.

**Последствия:**  
- Команде потребуется изучить особенности aiogram 3 (Router, FSM стратегии).  
- Мы получаем готовые инструменты для сложной логики и высокую производительность под нагрузкой.  
- Интеграция с Redis для хранения состояний и кэша будет бесшовной.
